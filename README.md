# Hermes BEAM

> A distributed, self-improving AI Agent framework built on Ash AI, Reactor, Nx/Bumblebee, and Tailscale — the BEAM takes on Hermes.

---

## What Is This?

**Hermes BEAM** is a private, sovereign AI agent network that runs entirely on your own hardware. It is inspired by the [Hermes Agent](https://yuv.ai/blog/hermes-agent) concept of autonomous, closed-loop learning — where an agent builds a deepening model of its user over time by creating, using, and self-improving its own skills and memory from experience.

Instead of relying on third-party APIs or cloud providers, Hermes BEAM runs on a home cluster of Mac Minis and a Gaming PC connected over a secure [Tailscale](https://tailscale.com) mesh VPN. The BEAM VM provides fault tolerance, hot code reloading, and effortless distributed computing that maps perfectly to the requirements of a durable, self-healing AI system.

---

## Core Design Philosophy

| Principle | Hermes (Original) | Hermes BEAM |
| :--- | :--- | :--- |
| **Memory** | Bounded text files (`MEMORY.md`, `USER.md`) | Bounded `Scratchpad` resource in PostgreSQL |
| **Skill Learning** | Python functions generated by the LLM | Elixir modules compiled dynamically on the BEAM |
| **Agent Loop** | Single-process Python script | `Reactor` workflow DAG (concurrent, with undo sagas) |
| **LLM Provider** | External API (Hermes 3 via NousResearch) | Local `Nx.Serving` cluster via Bumblebee (Llama 3, Phi-3) |
| **Distribution** | None (single machine) | Multi-node Erlang cluster over Tailscale |
| **Durability** | Restarts from scratch on crash | Ash Postgres persists all state; Reactor retries on failure |

### Node Roles

- **Hub Node (`NODE_TYPE=hub`):** Hosts PostgreSQL, acts as the central cluster coordinator via `libcluster_postgres`, runs the heavy 70B reasoning model, and executes scheduled global workflows.
- **Worker Node (`NODE_TYPE=worker`):** Runs smaller models suited to its Apple Silicon tier, executes Reactor workflows, and contributes compute to the mesh. Entirely stateless; data loss on restart is impossible.

## Hardware Topology

```
 NODE_ROLE=gaming_gpu    →  Tier 1 (70B reasoning) + Tier 2 (8B general)
 NODE_ROLE=mac_mini_pro  →  Tier 2 (8B general) + Tier 3 (3B docs)
 NODE_ROLE=mac_mini_base →  Tier 3 (3B docs)
```

Workloads are routed automatically. If an agent on a Mac Mini requires deep reasoning, Reactor dispatches the request to the Gaming PC's 70B model over the Tailscale tunnel via `Nx.Serving.batched_run/2`. No code changes are required.

## Key Components

### 1. Agent Memory (`Ash AI + pgvector`)

Memory follows a strict, bounded model inspired by Hermes. A `Scratchpad` resource maintains the agent's working memory and user profile as two character-limited strings (`memory_text` ≤ 2,200 chars, `user_text` ≤ 1,375 chars). The LLM is responsible for curating and condensing its own memory.

For historical context, a separate `Memory` resource stores vectorized interaction summaries, enabling semantic recall via pgvector's approximate nearest-neighbour search.

### 2. Agent Workflows (`Reactor`)

Every agent action is a `Reactor` workflow — a declarative, concurrent DAG of steps. Steps can:
- Run concurrently when independent.
- Compensate via `undo` functions, enabling automatic rollback on failure (saga pattern).
- Dynamically extend the graph at runtime by returning `{:ok, value, additional_steps}`.

### 3. Autonomous Skill Creation

The agent can generate Elixir code for repetitive tasks. Generated code is:
1. Stored as a `Skill` resource in PostgreSQL (persisted, auditable).
2. Compiled into a live BEAM module via `Code.compile_string/1`.
3. Registered as an Ash Action, making it available to future Reactor workflows.
4. On failure, a `undo` compensation triggers an LLM reflection prompt to rewrite the skill.

### 4. Cluster Discovery (`libcluster_postgres` + Tailscale)

Rather than broadcasting on the local network (Gossip) or requiring static IP configuration (EPMD), nodes register themselves in the central PostgreSQL database over the Tailscale mesh. When a node boots, it writes its presence; when it shuts down, it is automatically removed. All Erlang inter-node communication flows over WireGuard-encrypted Tailscale tunnels.

### 5. Model Serving (`Nx.Serving` + Bumblebee)

Each node loads only the models appropriate for its hardware. Models are defined by tier:

| Tier | Model | Hardware | Use Case |
| :--- | :--- | :--- | :--- |
| Tier 1 | Llama 3 70B | NVIDIA GPU | Deep reasoning, planning, reflection |
| Tier 2 | Llama 3 8B | Apple Silicon Pro / NVIDIA | General tasks, tool calling, synthetic data |
| Tier 3 | Phi-3 Mini | Apple Silicon Base | Documentation, formatting, JSON |

## Prerequisites

- **Elixir** ~> 1.16 and **Erlang/OTP** ~> 26
- **PostgreSQL** ~> 15 with the `pgvector` extension
- **Tailscale** installed and authenticated on all machines
- **EXLA** with either NVIDIA CUDA libraries (Gaming PC) or Apple Metal (`MPS`) support (Mac Minis)
- HuggingFace access token for gated models (Llama 3)

## Quick Start

### 1. Clone and install dependencies

```bash
git clone https://github.com/ShalokShalom/hermes-beam
cd hermes-beam
mix deps.get
```

### 2. Configure environment

```bash
# On every machine
export OPENAI_API_KEY=your_key_or_leave_empty_for_local_only
export DB_HOST=<Tailscale IP of your Hub machine>
export DB_USER=postgres
export DB_PASS=your_db_password
export DB_NAME=hermes_db
export HUB_IP=<Tailscale IP of your Hub machine>
```

### 3. Set up the database (Hub machine only)

```bash
export NODE_TYPE=hub
mix ecto.create
mix ecto.migrate
```

### 4. Boot the Hub

```bash
export NODE_TYPE=hub
export NODE_ROLE=gaming_gpu
bash bin/start_node.sh
```

### 5. Boot Worker Nodes (each Mac Mini)

```bash
export NODE_TYPE=worker
export NODE_ROLE=mac_mini_pro  # or mac_mini_base
export HUB_IP=100.x.x.x
bash bin/start_node.sh
```
